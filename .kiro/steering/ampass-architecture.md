# AMPass Project Architecture

## Overview

AMPass is a zero-knowledge encrypted password manager. The production app is PHP 8+ / MySQL running on cPanel or XAMPP. All vault secrets are encrypted client-side using Web Crypto API (AES-256-GCM) before being sent to the server. The server only stores ciphertext, IVs, salts, HMAC hashes, and Argon2id password hashes.

## Critical Security Rules

1. **NEVER store plaintext vault secrets** (passwords, card numbers, notes, keys) in the database or server logs.
2. **NEVER send decrypted vault item fields to the PHP server.** Encryption/decryption happens exclusively in the browser (or extension/desktop app).
3. **The server stores only:** encrypted_data (ciphertext), encryption_iv, title_hash (HMAC), url_hash (HMAC), password_strength (score), metadata flags.
4. **Master password** is sent to the server over HTTPS only for Argon2id hash verification. The actual key derivation for vault decryption happens client-side via PBKDF2.
5. **CSRF tokens** are required on all state-changing API calls.
6. **Rate limiting** is enforced on login, registration, vault unlock, and password changes.

## API Endpoints (Current)

All API routes are under `/api/{resource}/{action}`:

### Auth API (`/api/auth/`)
- `GET /api/auth/status` — Check login + vault unlock state, get CSRF token
- `GET /api/auth/derivationParams` — Get encryption salt, encrypted vault key, IV, iterations
- `POST /api/auth/verifyMaster` — Verify master password, unlock vault session
- `POST /api/auth/lock` — Lock vault

### Vault API (`/api/vault/`) — Requires vault unlocked
- `GET /api/vault/list` — List all items (encrypted blobs)
- `GET /api/vault/get/{id}` — Get single item
- `POST /api/vault/save` — Create or update item (receives ciphertext only)
- `POST /api/vault/delete` — Delete item
- `POST /api/vault/favorite` — Toggle favorite
- `GET /api/vault/stats` — Vault statistics
- `GET /api/vault/export` — Export encrypted backup (requires CSRF)
- `POST /api/vault/import` — Import encrypted backup

### Folder API (`/api/folder/`)
- `GET /api/folder/list` — List folders
- `POST /api/folder/save` — Create/update folder
- `POST /api/folder/delete` — Delete folder

### Share API (`/api/share/`)
- `POST /api/share/create` — Share item with another user
- `GET /api/share/list` — List shared items
- `POST /api/share/accept` — Accept a share
- `POST /api/share/revoke` — Revoke a share

## Encryption Architecture

1. **Registration:** Client generates random 256-bit vault key → derives wrapping key from master password via PBKDF2 (100k iterations) → encrypts vault key with AES-GCM → sends encrypted_vault_key + salt + IV to server.
2. **Unlock:** Client receives encrypted_vault_key + salt → derives wrapping key from master password → decrypts vault key → stores in JS memory + sessionStorage.
3. **Save item:** Client encrypts all fields as JSON blob with vault key (AES-256-GCM) → sends ciphertext + IV to server.
4. **Read item:** Client receives ciphertext → decrypts with vault key in memory.

## Technology Constraints

- **PHP production:** No Node.js, no build step, no framework. Plain PHP 8+ with PDO.
- **Extension/Desktop dev:** Node.js, TypeScript, Rust (Tauri) are allowed for development/build tooling only.
- **Never break:** The installer, the existing API, or the cPanel/XAMPP deployment.
- **HTTPS required** for production (Web Crypto API needs secure context).
